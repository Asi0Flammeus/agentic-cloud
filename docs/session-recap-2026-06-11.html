01

Diagnostic — l'anatomie du livelock

Symptôme ressenti : « ça rame, ça bug ». Réalité mesurée :

  • 4 vidéos d'avril (19.7 GB) existaient en local mais n'avaient jamais atteint crqpt:Videos/raw.
  • Chaque cycle bisync tentait l'upload ; TimeoutStartSec=300 le SIGTERM-ait en plein transfert → reprise à l'octet 0 toutes les 5 min, à l'infini.
  • Chaque kill laissait un lock rclone .lck empoisonné → les cycles suivants échouaient instantanément.
  • Uplink saturé en permanence → les mounts WebDAV (~/clouds/*) étouffaient : timeouts http2/TLS — c'était ça, le « rame ».
  • Aggravant : rec faisait écrire ffmpeg directement dans le nom final → le watcher inotify bisync-ait un fichier en cours d'écriture pendant les meetings.
racine conceptuelle
Des heuristiques calibrées pour un vault de 8 800 fichiers markdown (timer 60s, timeout 300s, bisync-everything) appliquées uniformément à des recordings multi-GB. Le mécanisme n'était pas cassé — il était mal assorti à la sémantique des données.

03

Implémentation — v2 en TDD

  • Champ strategy par pair (queue|mirror|bisync, défaut bisync = backcompat totale).
  • queue : push rclone copy --min-age 2m puis reconcile rclone sync remote→local — gated sur le succès du push, jamais sur un fichier <2 min (un enregistrement en cours est intouchable).
  • mirror : rclone sync local→remote avec fusible source-vide (un dossier local manquant ne peut pas vider le remote).
  • Self-heal : purge des locks rclone orphelins quand aucun process vivant ne les détient.
  • health.json + cloud sync health ; intervalle par pair respecté ; les pairs lourds passent en dernier.
  • Escalade : un seul message mx par épisode de panne, après 24h.
  • TimeoutStartSec 300s → 4h : un long transfert idempotent doit survivre ; l'anti-chevauchement appartient aux flocks.
  • Fix du clobber config au remount + --min-free configurable (était hardcodé 80G).
  • rec : enregistre en .ogg.partial + rename atomique au stop — le watcher ne voit plus que le fichier fini.
  • queue-trash : les suppressions du reconcile partent dans ~/.cache/agentic-cloud/queue-trash/ au lieu d'être détruites.

04

Incident live — le .ogg disparu

Pendant le déploiement, le .ogg de 13:13 a disparu du remote puis du local (reconcile fidèle). Enquête à chaud : ni l'envoy (playbook l'interdit), ni le sync (queue/mirror ne suppriment jamais le remote).

résolution
Confirmé par asi0 : c'était un test pour un envoy — suppression volontaire, comportement queue exactement comme designé. Le doute aura quand même accouché du filet queue-trash, qui reste en place : toute suppression reconcile est désormais récupérable localement.
05

Déploiement & vérification

  • Drop-in temporaire 2h posé en live → l'upload des 19.7 GB a enfin pu finir (27 min) ; lock expiré seul via --max-lock.
  • Config basculée, units systemd régénérées (4h baked-in, drop-in supprimé), watch relancé sur le code v2.
  • Listings bisync obsolètes purgés pour les pairs convertis.
  • Pill eww enrichi : pairs en échec + âge de la panne, fallback propre sans health.json.
  • Vérifié : cycle propre par pair sous sa nouvelle stratégie, intervalles respectés, flocks anti-chevauchement OK, 4/4 verts · 0 échec
06

Fils restants

  • Pré-chauffage VFS (« snappy comme du local ») : rclone rc vfs/refresh recursive après mount + périodique — designé dans docs/v2-redesign.md, non implémenté.
  • Cosmétique : le skip d'intervalle affiche « already running, skipped » — libellé trompeur, à distinguer.
  • La purge du queue-trash est manuelle pour l'instant (pas de rotation).
  • Quand le flux d'archivage envoy sera réel : valider le miroir de file en production (premier vrai raw→archive).
